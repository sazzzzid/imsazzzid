## Why I Built This

Most JavaScript tutorials explain closures with a paragraph and a diagram. VisualJS lets you **watch** a closure form: see the scope chain link, observe the outer variable survive garbage collection, watch the inner function capture its environment in real time.

This is not a video. It is a **custom JavaScript runtime** that instruments any code for step-by-step visual execution. [visualjs.in](https://visualjs.in) combines a structured curriculum (~85 chapters across 12 modules) with that engine. Everything runs in the browser. User code never leaves the device.

## High-Level Overview

VisualJS splits cleanly into a presentation layer (Next.js App Router) and a framework-agnostic core engine (`src/`). Chapter pages, the visualizer, and special tools sit in `app/`. Parsing, instrumentation, sandboxed execution, snapshot reduction, and pedagogy live in `src/`. Curriculum data and persisted user preferences connect both sides.

```
Presentation (Next.js)              Core Engine (src/)           Data
──────────────────────              ──────────────────           ────
Chapter pages (85 routes)    ──►    Parser (Babel AST)
/visualizer                  ──►    Instrumentation (__trace)
Special tools                ──►    Runtime (Web Worker)
Root layout (sidebar, SEO)   ──►    Snapshot (state reducers)
                             ──►    Pedagogy (explanations, why-engine)

app/_data/chapters           ◄── curriculum
app/_store (Zustand)         ◄── progress, theme, streaks
```

The Next.js layer handles routing, SEO, and learning UX. The `src/` engine handles parsing, execution tracing, state reconstruction, and pedagogical explanation. All client-side.

## The Execution Pipeline

This is the heart of VisualJS: how user code becomes visual, step-by-step state.

```
User clicks Run
    → executeCode(source)
    → validateCode() advisory lint
    → instrumentCode(): parse AST, inject __trace() calls
    → Web Worker sandbox (5s timeout, 10k event cap)
    → RuntimeEvent[] stream
    → SnapshotEngine.buildFromEvents()
    → applyEvent() per step (keyframe compression)
    → Snapshot[] timeline
    → explainStep(snapshot, prev) → human-readable narration
    → Rendered UI
```

### Instrumentation (`src/instrumentation/`)

User source transforms before execution by injecting `__trace()` calls at semantic boundaries.

```
Source Code
    ↓ @babel/parser
  AST (File)
    ↓ multi-pass @babel/traverse
  Visitors (per node type):
    variable-declaration, assignment-expression
    function-enter, function-call, return-statement
    if-statement, loop-statement, switch-statement
    async-function, await-expression
    class-declaration, prototype-access, this-binding
    try-statement, gc-tracking, module-tracking
    ↓ @babel/generator
  Instrumented Code + lineMap (generated→original line mapping)
```

Key file: `src/instrumentation/transforms/instrument.ts`.

```javascript
// What the user writes:
function greet(name) {
  const message = `Hello, ${name}!`;
  return message;
}
const result = greet("world");
```

```javascript
// What the engine executes (simplified):
__trace({ type: "function-enter", name: "greet", args: ["world"], line: 1 });
__trace({ type: "variable-declaration", name: "message", value: "Hello, world!", scope: "greet", line: 2 });
__trace({ type: "return", value: "Hello, world!", from: "greet", line: 3 });
__trace({ type: "function-exit", name: "greet", line: 4 });
__trace({ type: "variable-declaration", name: "result", value: "Hello, world!", scope: "global", line: 5 });
```

Instrumentation covers ES2024+ features: closures, prototypes, async/await, generators, classes, destructuring, and more.

### Runtime (`src/runtime/`)

**executor.ts** orchestrates lint, instrument, and worker dispatch.

**worker-executor.ts** runs sandboxed code in a Web Worker with no DOM, a 5-second timeout, and a 10,000-event cap.

**policy.ts** provides advisory code validation before execution.

Event modules split by domain:

- **events/core.ts:** variables, functions, loops, objects, console
- **events/async.ts:** promises, microtasks, macrotasks, await
- **events/advanced.ts:** generators, prototypes, classes, GC, modules

```typescript
// src/runtime/worker-executor.ts
const worker = new Worker("sandbox.js");

worker.postMessage({
  code: instrumentedCode,
  timeout: 5000,
  eventCap: 10_000,
});

worker.onmessage = (e) => {
  handleEvents(e.data.events); // RuntimeEvent[] flat stream
};
```

User code never runs on the main thread. Hard limits prevent infinite loops and memory exhaustion.

### Snapshot Engine (`src/snapshot/`)

Converts flat `RuntimeEvent[]` into navigable `Snapshot[]` for timeline scrubbing.

**State model** (`state-model.ts`): variables, scopes, call stack frames, heap objects (objects, arrays, functions), console output, async state (promises, microtasks, macrotasks), prototype chains, class instances, GC marks.

**Reducers** (`state-reducer.ts` delegates to domain sub-reducers):

- `reducers/variables.ts`
- `reducers/functions.ts`
- `reducers/control-flow.ts`
- `reducers/async.ts`
- `reducers/advanced.ts`

```typescript
type RuntimeState = {
  variables: Map<string, Variable>;
  callStack: StackFrame[];
  heap: Map<string, HeapObject>;
  scopes: ScopeChain;
  console: ConsoleEntry[];
  async: AsyncState;
};

function applyEvent(state: RuntimeState, event: RuntimeEvent): RuntimeState {
  switch (event.type) {
    case "variable-declaration":
      return variableReducer(state, event);
    case "function-enter":
      return functionReducer(state, event);
    case "function-exit":
      return stackPopReducer(state, event);
    // 30+ event types handled
  }
}
```

**Memory optimization:** keyframe compression every 64 steps plus LRU cache (max 16 replays). To jump to step 200: load keyframe at 192, replay 8 events.

## Closure Example

How VisualJS handles the classic closure gotcha:

```javascript
function createCounter() {
  let count = 0;
  return {
    increment: () => ++count,
    getCount: () => count,
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
console.log(counter.getCount()); // 2
```

Step by step in the visualizer:

1. `createCounter` executes and creates `count` in its scope
2. Two arrow functions are created and both capture the `count` binding
3. `createCounter` returns, but its scope survives (closure)
4. `increment()` modifies the captured `count`, visible in the heap graph
5. `getCount()` reads the same captured binding

Users scrub the timeline back and forth to watch this frame-by-frame.

## Async and Event Loop

```javascript
console.log("1: synchronous");

  console.log("3: macrotask");

  console.log("2: microtask");
});

```

## Key Architectural Decisions

**Engine/UI separation.** `src/` is framework-agnostic and testable in isolation. `app/` is Next.js presentation only.

**Event-sourced visualization.** UI reads Snapshot state, never raw AST at runtime.

**Worker sandbox.** All user code runs isolated with hard time and event limits.

**Keyframe snapshots.** Memory-efficient timeline for long executions (64-step keyframes).

**Route-per-chapter.** Each of 85 chapters is its own Next.js route for SEO, not one SPA.

**Colocated private folders.** `_components`, `_store`, `_hooks` per route keep features self-contained.

**Privacy-first.** No server-side code execution. User code never leaves the browser.

**Pedagogy as a layer.** Explanations computed from snapshots, not hardcoded per example.

**Babel over Acorn.** Full ES2024+ support for arbitrary user code.

**React Flow.** Graph visualization with pan, zoom, and custom nodes.

**No backend.** Zero ops cost. User owns their data.
